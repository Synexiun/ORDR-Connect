**Status:** Planned

## Executive Summary

Phase 6 delivered a fully operational web application: 27 pages, 28 UI components, 10 chart types, 4 API libraries, and 707 passing tests with zero ESLint errors under strictTypeChecked. The platform is visually and architecturally complete.

Phase 7 bridges the gap from a complete codebase to a live, revenue-generating product. The goal is to onboard 3 paying design partners from the collections / financial services vertical, validate the core value proposition under real operating conditions, and establish the feedback loops needed to reach General Availability.

Success in Phase 7 is binary: either 3 design partners are live and paying by the end of the phase, or the phase is not complete.

## Section 1 — Beta Infrastructure

### 1.1 Tenant Provisioning Flow

The platform must be able to create an isolated tenant environment in under 5 minutes without manual intervention.

**Required:**
- Self-serve signup form → WorkOS organization creation → tenant record in PostgreSQL → default RBAC roles provisioned → welcome email via SendGrid
- Manual provisioning path for design partners (internal admin tool or direct API call) as fallback during beta
- Tenant isolation verified at provisioning time: RLS policies active, `tenant_id` seeded into JWT claims, Redis ACL namespace scoped
- Provisioning event emitted to Kafka (`tenant.provisioned`) and logged to WORM audit trail

**Acceptance criteria:** A new tenant can be provisioned end-to-end (signup → first login) in under 5 minutes with zero manual steps.

### 1.2 Onboarding Wizard

First-time users must reach a working state without reading documentation.

**Required:**
- Step 1: Organization profile (name, industry, timezone)
- Step 2: Invite team members (email + role assignment)
- Step 3: Connect first data source (CSV import or API key for existing system)
- Step 4: Configure first channel (SMS via Twilio or email via SendGrid — credential entry + verification send)
- Step 5: Review compliance settings (HIPAA toggle, Regulation F limits for collections vertical)
- Progress persisted so wizard can be resumed; completion state tracked per tenant

**Note:** The `SampleDashboard` page and `demo/` pages already exist in the web app and can be surfaced during onboarding to show the product before real data is populated.

### 1.3 Demo Mode

Demo mode allows prospects to explore the platform with realistic, pre-seeded data — no account required.

**Required:**
- Isolated demo tenant (no real data, no outbound communications enabled)
- Realistic sample data: 500 synthetic customers, 2,000 interactions, 50 agent sessions, 10 active workflows — seeded from the existing demo pages
- Demo mode flag in JWT claims; all write operations return success but are no-ops against a shadow store
- Demo session time-limited to 60 minutes; conversion CTA presented at expiry
- Demo data refreshed nightly (Kubernetes CronJob)

**Security note:** Demo tenants must be fully isolated from production tenants at the database level (separate schema or RLS policy — not just application-layer gating). PHI controls apply even to synthetic data.

### 1.4 Feature Flags System

A feature flag system gates beta-only features and allows per-tenant capability toggling.

**Required:**
- Flag definitions stored in Redis with PostgreSQL as source of truth
- Evaluation: tenant_id + user_id + percentage rollout + explicit override
- Admin UI: toggle flags per tenant from the Settings page
- Flags evaluated server-side only — client never receives the raw flag definition
- Audit log entry created on every flag change (who changed what, when, previous value)
- Initial flag set: `beta_ai_agents`, `beta_marketplace`, `beta_partner_dashboard`, `beta_healthcare`, `beta_reports_advanced`

## Section 2 — Design Partner Program

### 2.1 Target Partners

Phase 7 targets **3 design partners** exclusively from the **collections / financial services** vertical. This is the first wedge market: highest ROI from AI automation, clearest compliance requirements (FDCPA, Regulation F), and lowest barrier to quantifying value (recovery rate, cost per contact).

**Ideal design partner profile:**
- Debt collection agency or financial services firm with 10–500 agents
- Currently spending $2,000–$20,000/month on CRM + dialer + compliance tooling
- Has an existing pain point with Regulation F contact attempt tracking
- Decision maker reachable (owner, VP Operations, or Compliance Officer)
- Willing to provide structured feedback bi-weekly for 90 days

**Target partner count:** 3 (minimum to validate, maximum to support deeply during beta)

**Compensation:** Design partners receive the Regulated tier ($199/user/month) at no cost for 90 days in exchange for structured feedback and permission to use as a reference customer.

### 2.2 BAA-Ready Onboarding (Healthcare Readiness)

While Phase 7 targets collections partners, the architecture must be ready to onboard healthcare partners in Phase 8. No additional engineering should be required at that point.

**Required before Phase 7 closes:**
- Business Associate Agreement (BAA) template finalized with legal counsel
- BAA acceptance flow in the onboarding wizard (checkbox + timestamp → WORM audit record)
- HIPAA toggle activates: field-level PHI encryption on all `customers` and `contacts` records for that tenant, PHI access logging enforced, minimum-necessary access policy applied
- BAA acceptance event emitted to Kafka and replicated to S3 Object Lock
- Breach notification workflow documented and tested (detect → assess → notify within 60 days)

### 2.3 In-App Feedback Collection

Structured feedback is captured inside the product — not via external surveys.

**Required:**
- Feedback widget: accessible from every page (floating button, bottom-right)
- Captures: page context, free-text, optional severity (blocker / friction / nice-to-have)
- Stored per tenant with timestamp and user identity
- Weekly digest emailed to internal team via SendGrid
- High-severity feedback (`blocker`) triggers Slack notification within 5 minutes
- Feedback data never leaves the tenant's data boundary (no third-party analytics tools during beta)

### 2.4 Usage Analytics and Telemetry

Usage data informs product decisions and surfaces partner health signals.

**Required:**
- Server-side event tracking only (no client-side analytics SDK during beta — avoids third-party data sharing)
- Events: page views (page name + tenant_id, no PII), feature usage (which components activated), agent actions triggered, errors encountered (type + frequency)
- Telemetry pipeline: application → Kafka (`telemetry.*` topic) → ClickHouse → Grafana dashboard
- Tenant health score: composite of login frequency, pages visited, agent actions executed, feedback submitted — surfaced in internal admin view
- Usage data classified as INTERNAL (not RESTRICTED); no PHI captured in telemetry

## Section 3 — Missing for Beta Readiness

The following items are currently unimplemented or mocked. All must be complete before the first design partner is onboarded.

### 3.1 Real API Integration

**Current state:** All web app pages use locally mocked data (hardcoded arrays, `setTimeout` simulations).

**Required:**
- Replace all mock data with calls to the real API (`/api/v1/*` endpoints)
- API base URL configured via environment variable (`VITE_API_BASE_URL`)
- Authentication header injected via Axios/fetch interceptor using the WorkOS JWT
- Loading states, error states, and empty states wired to real API responses
- React Query (or SWR) for cache management and background refresh
- Priority order: Dashboard → Customers → Interactions → AgentActivity → Analytics → remaining pages

**Estimated scope:** ~25 API integration points across 27 pages

### 3.2 Authentication Flow End-to-End

**Current state:** Auth is mocked in the web app (`apps/web/src/lib/auth.tsx`). The backend auth package is fully implemented.

**Required:**
- WorkOS `authkit-react` SDK integrated in the web app
- Login page wired to WorkOS-hosted login (redirect flow with PKCE)
- JWT stored in memory only (NOT localStorage, NOT sessionStorage — see Rule 2 and Rule 6)
- Refresh token rotation handled transparently
- MFA enforcement: users without MFA configured are redirected to MFA setup before accessing any protected page
- Session expiry: idle timeout (30 minutes) + absolute timeout (8 hours) with graceful re-auth
- Logout: server-side session revocation + WorkOS session termination + Redis session delete

### 3.3 Email Notifications

**Current state:** SendGrid is integrated in the `channels` package but no transactional emails are sent from the web application flows.

**Required transactional emails:**
- Welcome email on tenant provisioning (SendGrid dynamic template)
- Team member invitation (link valid 72 hours, single-use, signed JWT)
- Password reset / MFA setup prompt
- Agent action completed (for human-review-required actions)
- Weekly usage summary (partner health digest)
- Feedback acknowledgment (auto-reply within 2 minutes of submission)

**All emails must:**
- Use SendGrid dynamic templates (no HTML in application code)
- Include unsubscribe link (CAN-SPAM / GDPR compliance)
- Log send/delivery/bounce events to the WORM audit trail
- Never include PHI in subject lines or preview text

### 3.4 Webhook Event Delivery

**Current state:** Webhook infrastructure exists in the API (`/api/v1/webhooks`) but outbound webhook delivery to partner systems is not implemented.

**Required:**
- Outbound webhook delivery: partner registers URL + secret → events fan out on Kafka consumer
- Payload signed with HMAC-SHA256 (secret stored hashed in `api-keys` table)
- Retry policy: exponential backoff, 5 attempts over 24 hours
- Dead-letter queue for permanently failed deliveries (Kafka DLQ topic)
- Webhook delivery log in the developer console (event type, timestamp, response code, latency)
- Events available for webhook subscription: `customer.*`, `agent.*`, `interaction.*`, `compliance.*`

### 3.5 Stripe Billing Integration

**Current state:** The `billing` package implements metering and invoicing logic but has no Stripe integration.

**Required:**
- Stripe Customer created on tenant provisioning
- Subscription created at plan selection (Starter / Professional / Enterprise / Regulated)
- Usage-based billing: agent action overages reported to Stripe Billing Meters at end of each UTC day
- Stripe webhooks: `invoice.paid`, `invoice.payment_failed`, `customer.subscription.deleted` → Kafka events → application state
- Billing portal: self-serve plan upgrade/downgrade and invoice download via Stripe Customer Portal (embedded in Settings page)
- Design partner override: 90-day $0 subscription with expiry date tracked in tenant record
- PCI compliance: no card data ever touches ORDR-Connect servers (Stripe Elements / hosted fields only)

### 3.6 Production Deployment

**Current state:** Terraform modules and Kubernetes manifests are complete. No environment has been provisioned.

**Required:**
- Target: AWS `us-east-1` (primary) with `us-west-2` cross-region replication for audit logs
- Sequence:
  1. Terraform apply: networking → RDS → ElastiCache → MSK → Vault → EKS → monitoring → S3 WORM
  2. HashiCorp Vault initialized and unsealed; secrets seeded for all services
  3. Kubernetes: namespace → RBAC → NetworkPolicies → service deployments → Istio mesh
  4. DNS + TLS: Route 53 + ACM certificate for `app.ordr-connect.com`
  5. Smoke tests: all 92 API endpoints return expected status codes
  6. Canary deploy validation: 5% traffic → 100% traffic over 30 minutes
- Runbook documented in `infrastructure/runbooks/production-deploy.md`
- Monitoring: Grafana dashboards live, PagerDuty alert routing configured
- Baseline performance targets: p99 API latency < 500ms, p50 < 100ms
