**Date:** 2026-04-03
**Phases:** 50 (Zendesk + ServiceNow adapters), 51 (New agent types), 52 (Developer SDK polish)
**Status:** Approved (revised after spec review)

## Phase 50 — Zendesk + ServiceNow Adapters

### Goal
Close the integration gap: `packages/integrations` currently ships only Salesforce and HubSpot. Phase 50 adds Zendesk and ServiceNow as first-class CRM/ticketing adapters.

### Scope

**Files to create:**
- `packages/integrations/src/zendesk/adapter.ts` — implements `CRMAdapter`
- `packages/integrations/src/servicenow/adapter.ts` — implements `CRMAdapter`

**Files to modify:**
- `packages/integrations/src/types.ts` — add `'zendesk' | 'servicenow'` to `INTEGRATION_PROVIDERS` const array (TypeScript-only change; no DB migration required — no `integration_provider` Postgres enum exists; the integrations table is not yet in `packages/db/src/schema/`)
- `packages/integrations/src/index.ts` — export adapter factory keyed by provider
- `packages/integrations/src/__tests__/` — unit tests for both adapters

### Design

Both adapters implement the existing `CRMAdapter` interface (`packages/integrations/src/adapter.ts`) with no interface changes.

**Zendesk:**
- OAuth 2.0 via Zendesk's authorization code flow (`/oauth/authorizations/new`)
- Contacts → Zendesk Users, Deals → Zendesk Tickets, Activities → Zendesk Ticket Comments
- Webhook signature: HMAC-SHA256 on `X-Zendesk-Webhook-Secret-Header`
- Rate limit: 700 req/min (Zendesk Enterprise); surfaced via `getRateLimitInfo()`
- `bulkPushContacts()` implemented via Zendesk bulk user create/update API

**ServiceNow:**
- OAuth 2.0 via ServiceNow instance OAuth provider (`/oauth_auth.do`)
- Contacts → `sys_user` table, Deals → `incident` table, Activities → `sys_journal_field`
- Webhook signature: HMAC-SHA256 on `X-ServiceNow-Signature`
- Rate limit: 60 req/min (conservative default); surfaced via `getRateLimitInfo()`
- `bulkPushContacts()` implemented via ServiceNow Table API batch inserts

**Shared requirements (both adapters):**
- Credentials stored AES-256-GCM encrypted (via `packages/crypto`)
- All operations audit-logged via `packages/audit`
- Webhook signature verified before any payload processing (throws on failure)
- PHI field detection via `FieldMapping.isPhi` — encrypted before sync
- Normalized to `CrmContact / CrmDeal / CrmActivity` — no provider-specific types leak out

**Adapter factory (`index.ts`):**
```ts
createAdapter(provider: IntegrationProvider): CRMAdapter
// throws AppError if provider unknown
```

### Compliance
- SOC 2 CC6.1 — tenant-scoped, audit-logged
- ISO 27001 A.14.1.2 — TLS on all outbound calls
- HIPAA §164.312(e) — PHI encrypted in transit and at rest

## Phase 51 — New Agent Types

### Goal
Expand `apps/agent-runtime` from 3 built-in agents (support-triage, healthcare, escalation) to 5 by adding `sales_development` and `outbound_campaign`. The existing `churn_detection` role already covers churn scoring and retention scheduling — a separate `churn_prevention` role is not needed. Instead, a new `collections` agent is added (the `collections` role already exists in `AGENT_ROLES` but has no implementation).

### Scope

**Files to create:**
- `apps/agent-runtime/src/agents/sales-development.ts`
- `apps/agent-runtime/src/agents/outbound-campaign.ts`
- `apps/agent-runtime/src/agents/collections.ts`

**Files to modify:**
- `packages/core/src/types/agent.ts` — add `'sales_development'` and `'outbound_campaign'` to `AGENT_ROLES` array (note: `'collections'` already exists in `AGENT_ROLES`)
- `apps/agent-runtime/src/agent-registry.ts` — add `AgentConfig` entries for all three new agents
- `apps/agent-runtime/src/prompts.ts` — add prompt builders for each new agent

### Design

All three agents follow the established pattern:
1. Role string added to `AGENT_ROLES` in `packages/core/src/types/agent.ts`
2. `AgentConfig` created via `createAgentRole()` in `agent-registry.ts`
3. Prompt builder function in `prompts.ts`
4. Explicit tool allowlist (principle of least privilege) — all names in underscore_case matching registry keys
5. Confidence threshold ≥ 0.7 before autonomous action → else HITL queue
6. Full prompt → reasoning → action → outcome audit chain

**`sales-development.ts`**
- Role: `sales_development` — qualify inbound leads, book discovery calls, send initial outreach
- Tools: `lookup_customer`, `schedule_followup`, `send_sms`, `search_knowledge`
- NBA pipeline: score leads via `packages/decision-engine` before acting
- HITL gate: any outreach to a contact with no prior interaction requires human approval

**`outbound-campaign.ts`**
- Role: `outbound_campaign` — execute multi-step outbound sequences (SMS → email → call)
- Tools: `lookup_customer`, `send_sms`, `schedule_followup`, `summarize_conversation`
- Hard dependency on `packages/channels/consent.ts` — verifies opt-in before every send
- HITL gate: **mandatory before any send to >1 recipient** (flagged irreversible). The agent pauses after each batch of sends and awaits human approval before continuing. The session hard limit is 50 total sends; reaching 50 terminates the execution and requires a new human-initiated session.
- Action budget: `maxActions: 20` per execution (consistent with platform norms), not 50

**`collections.ts`**
- Role: `collections` — follow up on overdue accounts, check payment status, escalate unresponsive accounts
- Tools: `lookup_customer`, `check_payment`, `send_sms`, `escalate`, `schedule_followup`
- Trigger: account balance overdue ≥ N days (configured per tenant)
- HITL gate: escalation actions on accounts with outstanding balance > threshold require human review

### Compliance
- Rule 9 (Agent Safety): explicit tool allowlists, confidence thresholds, kill switch inherited
- All HITL gates logged as irreversible-flagged actions
- No PHI in agent prompts — tokenized references only

## Phase 52 — Developer SDK Polish

### Goal
Harden `packages/sdk` into a production-ready developer-facing toolkit and wire a submission flow into `apps/developer-portal`.

### Scope

**Files to modify:**
- `packages/sdk/src/agent-builder.ts` — ensure fluent API is complete and documented
- `packages/sdk/src/manifest-validator.ts` — `ValidationResult` typed errors (not raw Zod errors exposed)
- `packages/sdk/src/packager.ts` — add file I/O wrappers (additive, does not break existing in-memory API)
- `packages/sdk/src/test-harness.ts` — `TestReport` return type with budget utilization
- `packages/sdk/src/types.ts` — harden `AgentManifest` type

**Files to create:**
- `packages/sdk/src/schemas/agent-manifest.schema.json` — canonical JSON Schema for manifest validation

**Apps to modify:**
- `apps/developer-portal/` — SDK docs page + "Submit Agent" flow

### Design

**`AgentBuilder` (actual API — methods to verify/complete):**
```ts
const manifest = new AgentBuilder('my-agent')
  .withTool(tool: ToolDefinition)           // called once per tool (incremental)
  .withTool(anotherTool)
  .maxBudget(budget: AgentBudgetConfig)
  .withPromptBuilder(builder: SdkPromptBuilder)
  .confidenceThreshold(threshold: number)   // must be ≥ 0.7
  .build()                                  // throws ValidationError if invalid
```
No `withRole()`, no `withTools([])` batch method, no static `create()` factory — these do not exist. Constructor is `new AgentBuilder(name: string)`.

**`ManifestValidator`:**
- JSON Schema strict mode (`additionalProperties: false`) via `agent-manifest.schema.json`
- Validates: tool allowlist against known tools, budget within platform caps, confidence ≥ 0.7, required compliance fields
- Returns `ValidationResult` with typed `ValidationError[]` — not raw Zod/schema errors

**`Packager` (additive file I/O layer):**
- Existing API preserved: `packageAgent(agent: AgentPackage): Result<PackagedAgent>` (in-memory)
- Existing API preserved: `verifyPackage(pkg, original): Result<void>` (in-memory)
- **New additive functions:**
  - `packToFile(agent: AgentPackage, outputPath: string): Promise<Result<void>>` — calls `packageAgent` then writes `.ordr-agent` to disk
  - `verifyFromFile(archivePath: string, original: AgentPackage): Promise<Result<void>>` — reads archive, calls `verifyPackage`
- No breaking changes to existing functions

**`TestHarness`:**
- Sandbox runner: executes agent logic against injected mock tool responses
- Asserts output matches declared output schema
- Returns `TestReport`: `{ passed: boolean; assertions: AssertionResult[]; budgetUtilization: BudgetUtilization }`

**Developer Portal wiring:**
- New "Submit Agent" page: upload `.ordr-agent` archive → `verifyFromFile()` → manifest preview → submit to marketplace review queue
- SDK docs page: inline code samples using the actual `AgentBuilder` constructor API

### Compliance
- Rule 8 (Supply Chain): all submitted archives verified via SHA-256 + signature before marketplace listing
- Rule 9 (Agent Safety): `ManifestValidator` enforces confidence threshold and tool allowlist gates at submission time
- Rule 4 (Input Validation): manifest validated with strict JSON Schema before any processing

## Implementation Order

1. **Phase 50**: Zendesk + ServiceNow adapters (`packages/integrations`)
2. **Phase 51**: New agent types (`packages/core` → `apps/agent-runtime`)
3. **Phase 52**: Developer SDK polish (`packages/sdk` + `apps/developer-portal`)

Each phase is independently deployable. Phase 51 has an internal ordering constraint: `packages/core/src/types/agent.ts` must be updated before `agent-registry.ts`.
